# เอกสารข้อกำหนดสถาปัตยกรรมระบบ ฝั่งประมวลผลเซิร์ฟเวอร์ (Backend Architecture Specification - Admin Subsystem)

เอกสารฉบับนี้จัดทำขึ้นเพื่ออธิบายกลไกเชิงลึกและรายละเอียดของไฟล์ที่เกี่ยวข้องกับสิทธิการใช้งานระดับ **ผู้ดูแลระบบ (Admin Role)** เพื่อให้ทีมวิศวกรซอฟต์แวร์และผู้ดูแลระบบ ทราบถึงกระแสข้อมูล (Data Flow) โครงสร้างสถาปัตยกรรม และตรรกะระดับฝั่งเซิร์ฟเวอร์ (Backend Logic) ที่ควบคุมการทำงานของระบบ

---

## สรุปโครงสร้างไฟล์ที่เกี่ยวข้องกับระบบงานผู้ดูแลระบบ

การทำงานของระบบผู้ดูแลระบบถูกแบ่งออกเป็น 4 ส่วนประกอบหลัก ตามหลักการ **Separation of Concerns** ของเฟรมเวิร์ก FastAPI:

1.  **`app/api/deps.py`**: เลเยอร์ควบคุมและตรวจสอบสิทธิ์ (Authentication Dependency Layer)
2.  **`app/api/routers/admin.py`**: เลเยอร์ประมวลผลจุดเชื่อมต่อ (Controllers & Routing Layer)
3.  **`app/schemas/admin.py`**: เลเยอร์ตรวจสอบและจัดโครงสร้างข้อมูล (Data Validation & Schema Layer)
4.  **`app/models/*`**: เลเยอร์โครงสร้างและจัดการฐานข้อมูล (Relational Models Layer)

---

## 1. ระบบตรวจสอบสิทธิ์และการจัดการการเข้าถึง (Authentication & Authorization Layer)
**พิกัดไฟล์:** `app/api/deps.py`

ก่อนที่คำร้องขอ (Request) ใดๆ จะส่งถึงระบบประมวลผลของผู้ดูแลระบบ จะต้องผ่านการคัดกรองจากระดับ Dependency Injection ดังต่อไปนี้:

*   **`get_current_user`**: ฟังก์ชันสำหรับตรวจสอบสถานะเซสชันแบบรัดกุม โดยจะวิเคราะห์รหัสผ่าน (Token) พร้อมตรวจสอบคุณลักษณะ `status` ในฐานข้อมูลดั้งเดิม เพื่อยืนยันว่าผู้ใช้รายนั้นมีสถานะเป็น `ACTIVE` อยู่ กรณีที่ผู้ใช้ทำการออกจากระบบ (Logout) และกลายเป็น `INACTIVE` ระบบจะปฏิเสธการเข้าถึงด้วยรหัส `403 Forbidden` ทันที โดยมิต้องรอให้เซสชันของระบบหมดอายุ
*   **`RoleChecker(["Admin"])`**: ฟังก์ชันตรวจสอบบทบาท (Role-based Access Control) เมื่อวิเคราะห์ตัวตนผู้ใช้งานเสร็จสิ้น ระบบจะประเมินคุณลักษณะ `role` ว่ามีสิทธิ์เทียบเท่า "Admin" หรือไม่ หากระดับสิทธิ์ไม่สอดคล้อง ระบบจะปฏิเสธการประมวลผลทันที

---

## 2. ส่วนประมวลผลหลักและจัดการเส้นทาง (Core Controllers & Routing Layer)
**พิกัดไฟล์:** `app/api/routers/admin.py`

ไฟล์นี้ทำหน้าที่เป็น Controller หลักซึ่งรวบรวมตรรกะทางธุรกิจ (Business Logic) ที่มีความซับซ้อนสูงทั้งหมดของระบบ

### 2.1 ระบบแสดงผลข้อมูลสถิติรวม (`GET /admin/dashboard`)
*   **การประเมินการขยายตัวฐานข้อมูล (Cumulative Base Growth)**: ระบบจะดำเนินการสืบค้น (Query) เพื่อหาจำนวนเอกสารและจำนวนพนักงานที่ถูกสร้างขึ้นในระยะเวลาเดือนปัจจุบัน และนำไปเปรียบเทียบกับฐานข้อมูลรวมก่อนเดือนปัจจุบัน เพื่อหาอัตราส่วนของเปอร์เซ็นต์อัตราการเจริญเติบโต (Percentage Growth) ที่สะท้อนการเปลี่ยนแปลงที่แม่นยำ
*   **การแบ่งกลุ่มข้อมูลประทับเวลา (Time-series Aggregation)**: ใช้ความสามารถของโมดูล `calendar` วางเงื่อนไขสกัดรอบวิเคราะห์ช่วงเวลา เพื่อแย่งผลประกอบตารางสถิติออกเป็นรายสัปดาห์ (นับตั้งแต่วันจันทร์), รายเดือน และ ข้อมูลสถิติสะสมดั้งเดิมทั้งหมด (All Time)

### 2.2 ระบบบริหารบัญชีบุคลากร (User Management)
*   **`GET /admin/users`:** ประมวลผลดึงรายชื่อพนักงานทั้งหมด โดยซ่อนกลไกการนับยอดบุคลากรที่มีสถานะ `ACTIVE` แบบทันทีทันใด (Real-time Iteration) ออกมาแสดงเป็นภาพรวมสถิติ เพื่อส่งต่อให้ระบบแสดงผล
*   **`POST /admin/members`:** ระบบสนับสนุนการเพิ่มจำนวนทรัพยากรบุคคลโดยตรงผ่านผู้ดูแลระบบ (Administrative Execution) โดยทำงานร่วมกับการประมวลผลแฮชรหัสผ่าน (`get_password_hash`) เข้าฐานข้อมูลทันที 
*   **`DELETE /admin/users/{user_id}`:** กลไกลบทำลายข้อมูลรายการผู้ใช้และคำสั่ง `db.delete()` จากระบบฐานข้อมูลแบบถาวร

### 2.3 ศูนย์ควบคุมเอกสารและประเมินมาตรวัด (`GET /admin/documents`)
*   **การประมวลผลเปอร์เซ็นต์เชิงทิศทางเดียว (Monotonic Completeness Logic):** 
    แทนที่จะอ้างอิงสถานการณ์ทำงานโดยตรง ระบบทำการประมวลผลผ่าน 4 ระดับประเมินผลเชิงปริมาณ: (25%, 50%, 75%, 100%)
    ตรรกะหัวใจหลักคือระบบจะวิเคราะห์ประวัติศาสตร์โครงสร้างตาราง `doc.audits` เช็คการประทับตราอนุมัติรื้อถอน (`request_change_at`) หากเคยถูกตีกลับ ระบบจะตรึงเปอร์เซ็นต์ความคืบหน้าเชิงตรรกะไว้ที่ไม่ต่ำกว่า 75% อย่างเป็นทางการ แม้ว่าสถานะพฤติกรรมปัจจุบัน จะตกสู่สภาวะดำเนินการของเจ้าหน้าที่ระดับปฏิบัติงาน (Data Processor) ก็ตาม

### 2.4 ระบบวิเคราะห์และประเมินคอขวดสะสม (`GET /admin/work-tracking/summary`)
ตรรกะที่ซับซ้อนที่สุดของระบบคือ การเชื่อมโยงโครงสร้างภาระความรับผิดชอบ (Dynamic Responsibility Allocation)
*   **การจัดสรรผู้รับผิดชอบเชิงผันแปร (Dynamic Ownership Alignment):** 
    ระบบจัดสรรพฤติกรรมสอดคล้องกับค่า `doc.status`:
    *   เมื่อหน้างานยึดครองโดยพนักงานควบคุมข้อมูล (DC) (`DRAFT`, `REJECTED_OWNER`, `PENDING_AUDITOR`): ระบบจัดลำดับตารางให้แสดงชื่อแคมเปญหลักชั่วคราว พร้อมดึงรายชื่อพนักงาน DC ให้เป็นผู้รับผิดชอบบนกระดานระบบ
    *   เมื่อหน้างานยึดครองโดยพนักงานผู้ประมวลผล (DP) (`PENDING_PROCESSOR`, `REJECTED_PROCESSOR`): ระบบจะเจาะลึกเข้าไปในรายการแบบแผน `ProcessorRecord` ล่าสุด, กระทำการปรับเปลี่ยนเป้าหมายการแสดงผลชื่อเอกสาร และโอนโครงสร้างหนี้สินความรับผิดชอบไปแสดงเป็นชื่อพนักงานสายงาน DP
    *   แยกลำดับชี้วัดตัวแปรสิทธิผู้ตรวจสอบ (`auditor_name`) จัดวางแยกเป็นพารามิเตอร์อิสระ เพื่อมิให้รายงานโครงสร้างผู้ปฏิบัติงานปนกับผู้ตรวจสอบ
*   **การเปรียบเทียบหาเวลาบันทึกล่าสุด (Recursive Timestamp Tracking):** ระบบจะวนสืบค้นตรวจสอบลอจิกเวลาตารางเชื่อมโยงทั้งหมด เริ่มตั้งแต่ `created_at` พื้นฐาน ไปสิ้นสุดถึงโครงสร้างใบตรวจสภาพ Audit จากผู้ตรวจสอบ เพื่อคัดคั่นนำส่งค่าเวลาล่าสุด (`last_updated`) ที่แม่นยำที่สุดมาแสดงผล

## 3. เลเยอร์ตรวจสอบและจัดโครงสร้างข้อมูล (Data Validation Core)
**พิกัดไฟล์:** `app/schemas/admin.py`

เนื่องจากแพลตฟอร์ม FastApi ใช้ไลบรารี Pydantic ทำหน้าที่แกนกลาง เลเยอร์ส่วนนี้อธิบายข้อกำหนด (Specification) ของสถาปัตยกรรมข้อมูลหลัก:

*   **ควบคุมพารามิเตอร์ส่งออกและส่งเข้า (Strict Type Hinting):** มีการใช้อนุมานชนิดวิเคราะห์ UUID, String, List, Enum เพื่อป้องกันช่องโหว่ประเภทข้อมูลที่อาจส่งผิดพลาดจากระบบ Client
*   **การประกอบและห่อหุ้มโครงสร้างนำส่ง (API Payload Structuring):**
    *   `AdminDashboardStats`: กำหนดรูปแบบ Data Transfer Object (DTO) ที่รวบก้อนสถิติรวม อัตราความผันแปร และแผนภูมิเป็นโครงสร้างเดียว เพื่อลดปริมาณการร้องขอ (Request Count)
    *   `AdminUsersPageResponse` และ `WorkTrackingSummaryResponse`: ข้อกำหนดให้ผูกมัดค่าสถิติเชิงเดี่ยว (Scalar Statistics) ร่วมเข้ากับตาราง Array เพื่อเพิ่มประสิทธิภาพให้ Client ในการรับข้อมูลครบถ้วนต่อการเรียกใช้งานระบบเดี่ยว

## 4. เลเยอร์จัดโครงสร้างและเชื่อมโยงฐานข้อมูลเชิงสัมพันธ์ (Database Operations)
สถาปัตยกรรมทั้งหมดจะประมวลผลสถิติไม่ได้หากปราศจากความเชื่อมโยงระดับ Entity Relationship (ER) ควบคุมจากไฟล์เหล่านี้:
*   **`app/models/user.py`**: แหล่งระบุพพารามิเตอร์ Enums เช่น บทบาทผู้ปฏิบัติงาน (`UserRoleEnum`)
*   **`app/models/enums.py`**: ตัวแปรโครงสร้างสถานะระบบ บังคับกติกาการเซ็ตค่าควบคุมรัดกุม 100% 
*   **`app/models/document.py`**: ตารางศูนย์กลาง `RopaDocument` ผูกมัด Foreign Key แยกโครงสร้างเป็นความสัมพันธ์ 1-to-Many ร่วมกับตาราง `OwnerRecord`, `ProcessorRecord` และกระจายการเรียกใช้งานร่วมกับ `AuditorAudit` เพื่อตอบรับการดึงชุดประวัติศาสตร์ฐานข้อมูลลึกที่สุดด้วย Eager-Loading Framework ต่อไป
