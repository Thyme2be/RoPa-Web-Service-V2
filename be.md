# 🛑 สรุปรายการคำขอสำหรับทีม Backend (Backend Requirements)

เพื่อให้ระบบฝั่ง Frontend ทำงานได้อย่างสมบูรณ์ รบกวนทีม Backend ดำเนินการเพิ่มเติมในส่วนต่างๆ ดังนี้ครับ:

---

## 1. 📂 ระบบตารางเอกสารแอดมิน (Admin Document Table)
**สถานะปัจจุบัน**: ✅ แก้ไขแล้ว (Fixed - ตามสเปค Master API รายการที่ 1)

**สิ่งที่ต้องการ**:
- **Endpoint**: `GET /admin/documents`
- **Query Parameters**:
  - `search` (String): ค้นหาจากชื่อกิจกรรมหรือรหัสเอกสาร
  - `status` (String): กรองตามสถานะปัจจุบัน (`DRAFT`, `IN_PROGRESS`, `UNDER_REVIEW`, `COMPLETED`)
  - `page` (Int): เลขหน้าที่ต้องการเรียกข้อมูล
  - `limit` (Int): จำนวนรายการที่ต้องการต่อหนึ่งหน้า
- **รูปแบบการตอบกลับ (Response Build)**: ข้อมูลรายชื่อเอกสารที่มีฟิลด์ดังนี้:
  - `id`: รหัสเอกสาร (ตามรูปแบบ UUID หรือ ID ที่ส่งจากฐานข้อมูล)
  - `title`: ชื่อกิจกรรมการประมวลผล
  - `owner_name`: ชื่อผู้รับผิดชอบข้อมูลสังกัด
  - `department`: ชื่อแผนก
  - `dpo_name`: ชื่อเจ้าหน้าที่คุ้มครองข้อมูลส่วนบุคคล (ที่ได้รับมอบหมาย)
  - `updated_at`: วันที่อัปเดตล่าสุด
  - `status`: สถานะปัจจุบันระบบ (เช่น `COMPLETED`, `IN_PROGRESS`)

## 2. 📊 แดชบอร์ดรายบุคคล (Per-User Dashboard)
**เส้นทาง**: `GET /{id}/dashboard`

**สถานะปัจจุบัน**: ⚠️ ยังไม่แก้ (Still Buggy - Missing Imports และ **ไม่ตรงตามสเปค Master Sec 2**)

**ปัญหาที่พบ (CRITICAL BUG 🔥)**:
1. **NameError (Missing Imports)**: ❌ **ผลการตรวจโค้ดจริง**: ยังไม่ได้แก้ใน `dashboard.py` ทำให้ 500 Error
2. **Missing User Information**: ❌ **ผลการตรวจโค้ดจริง**: **ยังไม่มี** ออบเจกต์ `user` อยู่ที่ Top-level ตามที่สเปค Master Section 2 กำหนดไว้ (ปัจจุบันโค้ดส่งแค่ role_dashboard)
3. **Role Metrics Handling**: ตรวจสอบการรองรับบทบาท `ADMIN` และสถิติของ `OWNER/EXECUTIVE` ให้ครบถ้วน

**สิ่งที่ต้องการแก้ไข**:
- เพิ่มการ Import คลาสที่ขาดหายไปใน `dashboard.py` (บรรทัดที่ 28)
- ปรับโครงสร้าง Response ให้มีออบเจกต์ `user` คลุมด้านนอกสุดตามสเปค

## 5. 💬 Workflow ของ DPO และการส่งข้อเสนอแนะ

## 7. ✅ การยืนยันผลการตรวจสอบ โดย Auditor (Auditor Verification)
**เส้นทาง**: `PATCH /auditor/assignments/{id}/verify` (Endpoint ใหม่)

**สถานะปัจจุบัน**: ✅ แก้ไขแล้ว (Fixed - พร้อมใช้งานในระบบแล้ว)

**ผลการตรวจโค้ดจริง**: ไม่พบ Endpoint `PATCH` สำหรับการยืนยันในฐานข้อมูล
**ความต้องการ**:
- สร้าง Endpoint เพื่อรองรับการกด "ยืนยันการตรวจสอบ" จากฝั่งผู้ตรวจสอบ
- เมื่อมีการเรียกใช้ ให้ปรับสถานะใน `AuditorAssignmentModel` เป็น `VERIFIED` เพื่อจบขั้นตอนการตรวจสอบของผู้ตรวจสอบคนนั้นๆ

## 9. 🗑️ การอนุมัติการทำลายเอกสาร (Destruction Approval)
**สถานะปัจจุบัน**: ❌ **ยังไม่มีในระบบ (Missing PATCH Approval)**

**ผลการตรวจโค้ดจริง**: พบเพียง `GET` เพื่อดึงรายการ แต่สายงาน `PATCH` เพื่ออนุมัติหรือปฏิเสธยังไม่มีในทุก Router

**สิ่งที่ต้องการ**:
- สร้าง Endpoint เพื่อรับค่า `status` (`APPROVED` หรือ `REJECTED`) และ `rejection_reason`
- **Logic การทำงาน**:
  - เมื่อ `APPROVED`: ปรับสถานะคำขอทำลายเป็นอนุมัติ
  - เมื่อ `REJECTED`: ปรับสถานะเป็นไม่อนุมัติและบันทึกเหตุผลที่ได้จาก Frontend ลงในฐานข้อมูล
