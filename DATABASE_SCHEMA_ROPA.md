# 🗄️ โครงสร้างฐานข้อมูลและการจับคู่ (Database Schema Reference)

เอกสารฉบับนี้อธิบายโครงสร้างระบบฐานข้อมูลหลักเพื่อตอบสนองการทำใบการประมวลผลข้อมูลส่วนบุคคลแบบแยกสัดส่วนความรับผิดชอบ (Hybrid Data Entry) 

---

## 1. ผังโครงสร้างความเชื่อมโยงหลัก (Entity Relationship Core)

ในโปรเจกต์นี้จะปรากฏ 3 ตารางหลักทำงานร่วมกันเพื่อประกอบร่างเอกสาร RoPA ขึ้นมา 1 ฉบับ ได้แก่:

1. **`RopaDocument`** (เอกสารศูนย์กลาง)
   - เป็นตารางหัวใจหลักเสมือนกรอบรูป เสมือนกล่องเอกสาร 1 แฟ้ม
   - Data Owner เป็นเจ้าของโดยสมบูรณ์
   - จัดเก็บภาพรวมหลัก ๆ เช่น ลำดับ Versioning สถานะระดับบนสุด
2. **`OwnerRecord`** (ส่วนของผู้ควบคุม)
   - กระดาษเนื้อแท้ของ Data Owner ยึดติดกับกรอบรูปแบบ 1:1
   - ผังช่องกรอกข้อมูล 6 Section จะถูกประจุไว้ตรงนี้
3. **`ProcessorRecord`** (ส่วนของผู้ประมวลผล)
   - กระดาษแทรกพ่วงพิเศษที่สามารถปั๊มเพิ่มได้แบบ 1:N (1 เอกสาร มีหลาย Processor)
   - ทุกครั้งที่หัวหน้ากดมอบหมายงาน (Assign) ระบบจะสร้างกล่องใบนี้พ่วงติดขึ้นมา
   - ผังช่องกรอก 6 Section จะถูกประจุเหมือนของ Owner แต่อยู่ใต้ความรับผิดชอบของ Data Processor ท่ีถูกระบุชื่อไว้

---

## 2. โครงสร้างชุดสถานะควบคุม (Enum Control Status)

### 2.1 สถานะของ `RopaDocument` (DocumentStatus)
สถานะส่วนนี้มีไว้เพื่อให้ Data Owner และ Auditor รับทราบจังหวะเข้าทำประตูของเอกสารภาพรวมทั้งหมด:
- `DRAFT`: เริ่มสร้างแล้ว อยู่ระหว่างการเติมเต็มข้อความให้สมบูรณ์
- `PENDING_PROCESSOR`: กระจายเอกสารแนบไปยังโต๊ะของลูกน้องแล้ว รอให้ลูกน้องจัดส่งกลับมา (สถานะนี้ยังรอการยืนยันใช้งานชัดเจน)
- `PENDING_AUDITOR`: ประกอบกระดาษครบทุกใบ Owner ได้ประทับตรายืนยันแล้ว นำขึ้นโต๊ะให้ Auditor พิจารณา
- `APPROVED`: Auditor สำรวจแล้ว และตีตราประทับผ่าน
- `REJECTED_OWNER`: Auditor แทงกั๊กเอกสาร พบข้อบกพร่องในส่วนข้อความของ Data Owner
- `REJECTED_PROCESSOR`: Auditor แทงกั๊กเอกสาร พบข้อบกพร่องในส่วนข้อความของ Data Processor

### 2.2 สถานะของ `ProcessorRecord` (ProcessorStatus)
สถานะในโต๊ะทำงานของลูกน้อง ซึ่งไม่มีความเกี่ยวข้องกับเอกสารภาพรวม แต่ตารางหน้าบ้าน (Sidebar 1 ของ DP) ควรอ้างอิงสถานะนี้ 100%:
- `PENDING`: ออเดอร์งานเพิ่งโผล่ลงบนโต๊ะลูกน้อง
- `IN_PROGRESS`: ลูกน้องกดคีย์บอร์ดพิมพ์ตัวอักษรบางส่วน และเซฟ Draft ไปบ้างแล้ว
- `CONFIRMED`: ลูกน้องประกาศกร้าวว่าข้อมูลช่องบังคับทั้งหมดเติมเต็มและพร้อมส่งตรวจสอบ
- `SUBMITTED`: นำส่งเอกสารออกจากโต๊ะตนเอง ปลายทางกลับไปยังกระเป๋าเอกสารของ Owner
- `NEEDS_REVISION`: ลูกโป่งแตก Auditor ชี้เป้าว่าต้องแก้

## 3. รูปแบบพิเศษและกฎกติกาการเก็บตัวแปร 

### 3.1 การเก็บ Arrays แบบ Text Constraints
ตัวแปรบางประเภทที่หน้าบ้านรับเข้ามาเป็นรายการตัวเลือกลากคลุม (Multiple selections) เช่น `personal_data`, `data_category`, `retention_method`, `retention_storage_type`
- **Frontend Input:** `["รายการ A", "รายการ B"]`
- **Database Storage:** ระบบแปลง List สอดแทรกให้กลายเป็นรูปแบบ JSON String แบบดิบๆ เป็น `VARCHAR` / `TEXT` เพื่อประหยัดพื้นที่คอลัมน์ตาราง
- **API Response:** ก่อนที่ API จะสลิปข้อมูลคืนให้ Frontend ระบบจะควัก String แปลงสลับคืนให้กลายเป็น Array รูปเดิมให้ ทำให้ Frontend สบายดั่งเดิมไม่ต้องนั่งจับ Parse JSON บนฝั่งของตัวเอง

### 3.2 กฎหมายบังคับกิ่งตัวแปรเรื่องต่างประเทศ (Transfer Dependencies)
สำหรับคอลัมน์ `transfer_is_transfer` (มีการโอนข้อมูลส่วนบุคคลไปนอกประเทศหรือไม่)
หากค่าตัวแปรเป็น `true` ฐานข้อมูลและระบบ Validation จะคาดหวังให้มีการกรอกข้อมูลต่อไปนี้แบบหลีกเลี่ยงไม่ได้ (Required):
- `transfer_country`
- `transfer_method`
- `transfer_protection_std`
- `transfer_exception`
หากตัวแปรดังกล่าวถูกปรับเป็น `false` ตัวแปร 4 ชุดด้านบนสามารถส่งมารูปแบบของว่าง (`null`) ก็ถือว่าระบบพยักหน้าอนุญาตให้ผ่าน

### 3.3 ระบบห่อข้อความ Feedback จาก Auditor
ตารางโครงผลการตรวจสอบ `AuditorAudit` มีคอลัมน์สำคัญคือ `processor_feedback` และ `owner_feedback`
แทนที่จะรวมคอมเมนต์ก้อนเดียวเป็น String เดี่ยวๆ เรากำหนดให้เขียนลงเป็นก้อน **JSON String Directory** เช่น:
```json
{
  "section_1": "เบอร์ติดต่อของหัวหน้าคลาดเคลื่อน",
  "section_5": "หลักการส่งข้อมูลออกต่างประเทศยังแปลกๆ"
}
```
เพื่อให้หน้าบ้านสามารถจัดหมวดหมู่ แจวกระดานคำเตือน ไปแปะใต้ฟอร์ม Input ได้อย่างถูกต้องตรงเป้าหมายที่สุด (Pin-point Error System)
